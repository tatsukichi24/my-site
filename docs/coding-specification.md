## 4. レスポンシブ・レイアウト

### 4.1 方針

- **PC ファースト**でコーディングする。
- 対応ビューポート幅：**375px ～ 1920px**（最小 375 を下回らない前提で検証）。

### 4.2 ブレークポイント

メディアクエリは次の幅を基準とする（`max-width` 併用の場合は設計に合わせて調整可）。

| 名称 | 条件（例） | 用途の目安 |
|------|------------|------------|
| PC ワイド | **1920px 以上** | レイアウトは **vw で比例拡大**（頭打ちなし・下記） |
| PC | 1041px ～ 1919px | 中間〜広め PC（vw スケール） |
| タブレット | 769px ～ 1040px | タブレット横・小型ノート等 |
| SP | **768px 以下** | スマートフォン（375 含む）。レイアウトトークンは `100%` + 固定ガターへ切替 |

### 4.3 1920px デザイン基準のレイアウト幅（実装仕様）

- **デザイン幅は 1920px**。ページごとにコンテンツ幅が異なる（例: TOP 1400px、サービス 1500px、会社概要 1600px 等）。
- **レイアウト幅**は `assets/css/foundation/variables.css` の `--jgs-layout-*` トークン（`値 / 1920 * 100vw`）で管理する。
- **1920px 超**も `max-width` で頭打ちせず、**vw 比例拡大を継続**する。
- **タイポグラフィ**は既存の `clamp(min, vw, max)`（1920px 上限）を維持し、4K 等で文字が過大にならないようにする。
- **768px 以下**はレイアウトトークンを `100%` に上書きし、`--jgs-gutter`（16px）で余白を取る。

主なレイアウトトークン（1920px 時）:

| トークン | 幅 | 主なページ |
|---------|-----|-----------|
| `--jgs-layout-1400` | 1400px | TOP、取扱商品 |
| `--jgs-layout-1500` | 1500px | 店舗一覧、サービス内容 |
| `--jgs-layout-1600` | 1600px | 会社概要、採用系（左ナビ含む） |
| `--jgs-layout-1100` | 1100px | お知らせ一覧 |
| `--jgs-layout-1300` / `--jgs-layout-800` | 1300 / 800px | お知らせ詳細（外枠 / 記事） |
| `--jgs-layout-1000` | 1000px | プライバシーポリシー |
| `--jgs-layout-900` | 900px | お問い合わせ、エントリー |

## 8. CSS（クラス命名・ファイル分割）

### 8.1 命名（BEM）

- **BEM**（Block Element Modifier）を採用する。
- 例: `.card`、`.card__title`、`.card__title--featured`
- WordPress 化後もテーマ CSS との衝突に注意し、必要に応じてプレフィックス（例: `.jgs-`）を検討する。

### 8.2 単一ファイルにまとめない

- **すべてのスタイルを 1 つの CSS に書かない。** 役割とページ単位でファイルを分け、保守・レビュー・WP 化時の切り出しを容易にする。
- 分割の考え方は **§14.2** に従う。

### 8.3 読み込み方法（ビルドなし）

- 各 HTML から **`<link rel="stylesheet">` を複数行で読み込む**（`@import` の多用はレンダリングを遅らせることがあるため、原則 `<link>` を推奨）。
- **記述順がカスケードの前提になる。** 次の順を守ること:  
  **foundation → layout → components（共通で必要なもののみ）→ pages（そのページ専用）**
- 全ページ共通の束が多すぎてリクエスト数が気になる場合は、運用で「共通 2〜3 本」程度にまとめる判断は可。その場合も **ソース上のコメントや本書で、論理分割（どのブロックがどのファイル由来か）** を分かるようにする。

### 8.4 カテゴリ・サービス別の CSS

- **レイアウトが同一で装飾のみ違う**場合: 1 ファイル（例: `pages/products-category.css`）に BEM の Modifier で吸収する。
- **カテゴリ・サービスごとにセクション構成やコンポーネントが大きく違う**場合: `pages/products/` 配下に `category-{slug}.css` のように **ファイルを増やし、該当ページの HTML だけで追加読み込み**する。

## 12. アクセシビリティ（補足）

現時点で詳細要件は未定のため、以下を**最低限の推奨**として実装する。

## 14. ディレクトリ構成・CSS 分割

静的フェーズの**推奨構成**である。実際のスラッグは案件で置き換える。

### 14.1 プロジェクト全体（HTML の置き場所）

**ページ種別ごとにディレクトリを分け、各フォルダに `index.html` を置く**（クリーン URL を想定: `/products/` → `products/index.html`）。  
アセットへの参照は **ルート相対**（例: `/assets/css/...`）を推奨し、階層が変わってもパスを統一しやすくする（ローカルは簡易サーバーでルートをプロジェクトに合わせる）。

```
（プロジェクトルート）/
├── docs/
│   └── coding-specification.md
├── assets/
│   ├── css/                    … §14.2 参照
│   ├── js/
│   │   ├── common.js           … 全ページ共通（ヘッダー等）
│   │   └── pages/              … ページ固有が必要なら（例: top.js）
│   ├── image/
│   └── svg/
├── index.html                  … TOP
├── products/
│   ├── index.html              … 取扱商品一覧
│   └── {category-slug}/        … カテゴリごと（例: rings, necklace）
│       └── index.html
├── services/
│   ├── index.html              … サービス一覧
│   └── {service-slug}/
│       └── index.html
├── stores/
│   └── index.html              … 店舗紹介
├── news/
│   ├── index.html              … お知らせ一覧
│   └── {slug}/                 … 詳細（静的サンプル用。WP 化後はテンプレート）
│       └── index.html
├── company/
│   └── index.html
├── recruit/
│   └── index.html
├── contact/
│   └── index.html
└── privacy/
    └── index.html
```

| 公開 URL（想定） | ファイル |
|------------------|----------|
| `/` | `index.html` |
| `/products/` | `products/index.html` |
| `/products/{category}/` | `products/{category}/index.html` |
| `/services/` | `services/index.html` |
| `/services/{service}/` | `services/{service}/index.html` |
| `/stores/` | `stores/index.html` |
| `/news/` | `news/index.html` |
| `/news/{slug}/` | `news/{slug}/index.html` |
| `/company/` | `company/index.html` |
| `/recruit/` | `recruit/index.html` |
| `/contact/` | `contact/index.html` |
| `/privacy/` | `privacy/index.html` |

WordPress 化時は、上記の論理構造に合わせて **テンプレートファイル・カスタム投稿タイプ**へマッピングし、`assets` はテーマ配下へ移行する。

### 14.2 CSS ディレクトリ（ファイル分割）

`assets/css/` 配下を **レイヤー別**に置く。**1 ファイルに全スタイルを集約しない。**

```
assets/css/
├── foundation/
│   ├── variables.css           … :root の CSS 変数、フォント定義
│   ├── reset.css             … リセット（使用する場合）
│   └── base.css              … body、リンク、見出し共通タイポなど
├── layout/
│   ├── header.css
│   ├── footer.css
│   └── layout.css            … メインラッパー・グリッド共通
├── components/
│   ├── button.css
│   ├── card.css
│   ├── breadcrumb.css
│   └── …                     … 複数ページで使う UI パーツを追加
└── pages/
    ├── top.css
    ├── products-list.css
    ├── products-category.css   … カテゴリ詳細が近い場合は共有
    ├── services-list.css
    ├── services-detail.css
    ├── stores.css
    ├── news-list.css
    ├── news-detail.css
    ├── company.css
    ├── recruit.css
    ├── contact.css
    ├── privacy.css
    └── products/               … カテゴリ別に大きく異なる場合のみ
        └── category-{slug}.css
```

**各 HTML での読み込み例（順序重要）:**

1. `foundation/variables.css` → `foundation/reset.css`（任意）→ `foundation/base.css`
2. `layout/header.css` → `layout/footer.css` → `layout/layout.css`
3. そのページで使う `components/*.css` のみ
4. 該当する `pages/*.css`（および必要なら `pages/products/category-xxx.css`）

## 付録 A: CSS の「種類」について（簡易説明）

コーディングでよく出る言葉の整理です。**本案件は BEM + 素の CSS を採用**（ビルドなし）とする。

| 用語 | 説明 |
|------|------|
| **素の CSS（本案件）** | 拡張子 `.css` をそのままブラウザが解釈。追加ツール不要。色や余白は **ネイティブの CSS 変数**（`:root { --color-primary: ... }`）で整理可能。 |
| **SCSS / Sass** | CSS に変数・入れ子・ミックスインなどを足した言語。`.scss` をビルドして `.css` に変換してから公開することが多い。**本案件では採用しない。** |
| **PostCSS** | CSS を変換するツールチェーン。自動ベンダープレフィックス等に使う。**本案件では採用しない**（必要なプレフィックスは手書きまたは限定的に対応）。 |
| **Tailwind CSS** | HTML にユーティリティクラスを並べる手法。ビルドが前提。**本案件では採用しない。** |
| **CSS Modules** | 主にコンポーネント型フレームワーク向け。**本案件では採用しない。** |

## 付録 B: 構造化データ（含める場合の例）

ページ公開時に、以下のような JSON-LD を `Organization` として埋め込むことを想定する（内容は実データに差し替え）。

```html
<script type="application/ld+json">
{
  "@context": "https://schema.org",
  "@type": "Organization",
  "name": "ジュエルグランスエムネ",
  "url": "https://example.com/"
}
</script>
```

店舗・商品・パンくずなどは、ページ種別に応じて `LocalBusiness`、`Product`、`BreadcrumbList` 等を追加検討する。
